# Milestone 10 — Domain Operations Layer (Entity Management)

## Mission

Expose Domain Operations through the Application layer and CLI so a user
can fully populate PAIOS with commands instead of editing JSON. No
intelligence, no AI, no dashboard, no REST API, no GUI, no Scheduler /
Decision Engine / Runtime changes — only existing domain concepts, exposed
cleanly.

## 1. Architecture

### New component

`paios.application.domain_operations.DomainOperations` — one class holding
every entity-management operation. Each operation is pure orchestration:

1. construct the aggregate through its domain constructor, or load it
   through a repository interface;
2. mutate it only through documented aggregate methods
   (`Goal.accept`, `Goal.change_status`, `Progress.update`,
   `Principle.reviewed`, `Resource.consume/produce`, `Knowledge.revise/
   mark_applied`, `Event.link_reflection`);
3. persist through the repository interface (`save` / `update`).

No decision, no policy, no validation lives in the component — the domain
validates, the repositories persist.

Construction (in `bootstrap.build_components`) injects two things:

- the repository provider (`RepositoryFactory`, seen only through a local
  `OperationsRepositoryProvider` Protocol that names repository
  **interfaces** exclusively);
- `now: Callable[[], datetime]`, wired to the composition-root Clock —
  Domain Operations never imports the Runtime's Clock type, so the
  mission's dependency rule holds structurally.

The `Application` facade gains one delegation method per operation
(unchanged facade philosophy: composition and delegation only, zero
logic). `Components` gains the `operations` field.

### Runtime interaction (deliberate non-interaction)

Entity management goes `application → repositories.interfaces → domain`,
**bypassing the Runtime by design** (the mission's dependency graph and
"No Runtime modifications"). The Kernel's repository access is confined to
its boot sequence (approved resolution C5), so aggregates written here
enter Runtime State at the next boot. Consequences:

- All M10 `list` / `show` commands read the store through the operations
  layer, so they are always fresh — including inside a long-lived shell.
- Runtime views (`snapshot`, `status` aggregate counts) reflect newly
  created entities after the next start. One-shot CLI commands compose,
  start, execute, and stop per process, so in practice every one-shot
  invocation sees everything.

### Archive resolution (design decision, flagged for review)

The mission's example CLI shows `goal archive 1`, but the canonical domain
defines no Archived state for Goal or Project — `ENTITY_RELATIONSHIPS.md`
fixes Project status as `Active/Completed/Paused`, `enums.py` (frozen M1
baseline) matches, and Archived exists only for Event, Context Window, and
Event Disturber. Implementing `goal archive` would have required modifying
the frozen Domain Layer, contradicting both the audit rule ("No
frozen-layer modification") and the mission's own scope ("Only expose
existing domain concepts cleanly"). Resolution:

- Goal and Project expose their **canonical** lifecycle:
  `complete` / `pause` / `resume` (via `change_status`).
- Archive is exposed where the domain defines it: the new `archive-event`
  command delegates to the already-existing facade/Scheduler
  `archive_event` path (no Scheduler change).
- "Never mutate archived entities" is enforced — and tested — through the
  Event aggregate: no transition leaves Archived, and only immutable
  evidence (a Reflection link) may still be recorded.

If an Archived state for Goal/Project is wanted, that is a Domain Layer
change requiring an explicit approved resolution in a future milestone.

### Duplicate detection

Aggregate identifiers are freshly generated UUIDs, so identifier collisions
cannot express "I already created this". The operations layer therefore
guards uniqueness with a `find_by` attribute-equality query (a persistence
query, not a business rule) and raises `DuplicateEntityError`:

| Aggregate | Uniqueness criterion |
| --- | --- |
| User, Goal, Project, Principle, Context | `name` |
| Resource | `(user_id, type)` |
| Knowledge | `(user_id, domain, topic, concept)` |
| Reflection | at most one per Event — enforced by the **domain** (`Event.link_reflection`) |

### Read-only aggregates

Habits and Insights are generated by the Learning Engine. They get
`list` / `show` only; no create or mutate operation exists anywhere in the
Application API or CLI (`habit add` is an unknown command).

### Notable orchestration choices

- **`goal add` accepts immediately.** DOMAIN_MODEL.md Principle 4 says
  Goals require user acceptance; when the user states a goal directly, the
  statement *is* the acceptance, so the operation constructs the Goal with
  `suggested_by="User"` and calls the documented `accept()` method.
  `goal accept` remains available for engine-suggested Goals.
- **`project add` creates the Project's Progress.** Progress belongs to
  exactly one Project (BUSINESS_RULES.md); creating them together and
  attaching via `attach_progress` keeps `project progress` meaningful from
  birth.
- **`resource consume` / `resource produce`** realize the example's
  `resource update`: the domain models Resources as flows, not settable
  values, so direct value assignment would bypass aggregate methods.
- **`reflection add`** loads the Event, builds the Reflection from the
  Event's own Context Window (a Reflection cannot exist independently),
  and calls `Event.link_reflection` **before** persisting anything, so a
  domain rejection leaves the store untouched.
- **CLI parser additions are purely syntactic**: two-token command names
  (`goal add`) and shlex-based quoting (`goal add "Learn Sanskrit"`); the
  one-shot entry point re-quotes argv tokens so OS-shell grouping
  survives. The parser still knows names, arities, and usage — never
  meanings.

## 2. Dependency graph

```
cli  (parse, resolve refs, format)
 │      one primary delegation per command
 ▼
application  (facade → DomainOperations; orchestration only)
 │      OperationsRepositoryProvider protocol
 ▼
repositories.interfaces  (Repository[E, I] contracts)
 │
 ▼
domain  (entities validate; aggregate methods mutate)
```

- `domain_operations.py` imports: `paios.domain.*`,
  `paios.repositories.interfaces`, `paios.application.exceptions`, stdlib.
  Nothing from runtime, scheduler, decision_engine, learning, or
  infrastructure.
- CLI imports only `paios.cli.*`, the `Application` facade, and domain
  enums/identifier types for input parsing (the established M8
  convention).
- Time flows in as an injected callable; storage is reached only through
  interfaces. The dependency arrow never reverses.

## 4. Application API

All methods below are pure delegations from the `Application` facade to
`DomainOperations` (signatures abbreviated; identifiers are typed IDs):

```
add_user(name) -> User                      list_users() / get_user(id)
add_goal(user_id, name, description="")     list_goals() / get_goal(id)
accept_goal(id)  complete_goal(id)  pause_goal(id)  resume_goal(id)
add_project(user_id, name, description="")  list_projects() / get_project(id)
get_project_progress(id) -> Progress|None
update_project_progress(id, completion_percentage)
complete_project(id)  pause_project(id)  resume_project(id)
add_principle(name, category, description="")
list_principles() / get_principle(id) / review_principle(id)
add_resource(user_id, type, current_value, unit, negative_allowed=False)
list_resources() / get_resource(id)
consume_resource(id, amount) / produce_resource(id, amount)
add_context(name, location=None, people=(), emotion=None, trigger=None,
            reason=None, environment=None, notes=None)
list_contexts() / get_context(id)
add_knowledge(user_id, domain, topic, concept, project_id=None,
              difficulty=None, confidence=0.0, source=None)
list_knowledge() / get_knowledge(id)
revise_knowledge(id, confidence=None) / apply_knowledge(id)
add_reflection(event_id, facts=None, interpretation=None, root_cause=None,
               lesson_learned=None, improvement=None, confidence=None)
list_reflections() / get_reflection(id)
list_habits() / get_habit(id)               (read-only)
list_insights() / get_insight(id)           (read-only)
```

New exceptions: `DuplicateEntityError`, `ProgressNotAttachedError`
(both `ApplicationError` subclasses). Domain and repository errors
propagate unchanged (`DomainValidationError`, `InvariantViolationError`,
`ImmutabilityViolationError`, `EntityNotFound`).

## 6. Tests

`tests/application/test_domain_operations.py` (40 tests) and
`tests/cli/test_entity_commands.py` (24 tests):

- **Application operations** — add/list/get, lifecycle statuses, goal
  acceptance (user-created and engine-suggested), progress
  create-and-update, principle review immutability, resource flows.
- **CLI commands** — end-to-end over a real started application: every
  noun-verb command, quoted names, `field=value` tails, `<ref>`
  resolution, formatted output.
- **Repository / restart persistence** — every managed aggregate written,
  application stopped, a fresh application (and the real one-shot
  `main()` across processes) sees identical state; the rebooted kernel
  loads new aggregates into Runtime State.
- **Invalid operations** — unknown ids (`EntityNotFound`), operations
  before `start()`, non-numeric CLI amounts, unknown enum values, unknown
  or malformed `field=value` keys, unbalanced quotes.
- **Archive behavior** — Completed → Archived via `archive-event`; no
  transition leaves Archived; a Reflection link (immutable evidence) is
  still accepted.
- **Duplicate detection** — per-aggregate uniqueness criteria raise
  `DuplicateEntityError`; a second Reflection on the same Event is
  rejected by the domain as an immutability violation.
- **Entity validation** — blank names, out-of-bounds
  completion/confidence, negative resource values and non-positive
  amounts all rejected by the **domain**, surfacing unchanged through the
  facade.
- **Read-only** — `habit add` / `insight add` are unknown commands; no
  create operation exists on the facade or operations component.
- **Regression** — full suite: **596 passed** (531 pre-existing + 64 new;
  one M8 parser test loosened from set-equality to superset so the
  original command registry remains asserted verbatim while M10 commands
  may exist).
